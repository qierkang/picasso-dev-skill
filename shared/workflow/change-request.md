# 需求变更工作流

## 目标

基于已有需求基线，明确变更范围、影响点和回归范围。

## 变更分级

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| `L1` 轻微 | 文案修正、UI 微调、字段别名 | 记录变更日志后继续 |
| `L2` 一般 | 非核心字段、校验、枚举或公共规则调整 | 回退受影响文档并重新确认 |
| `L3` 重大 | 数据模型、核心流程或多模块联动变更 | 先做影响分析，再决定拆新任务或整体回退 |

## 标准阶段

1. 判断当前任务能力：`docs / dev / db / deploy`
2. 进入编码、SQL、启动前先运行对应 `doctor.sh`
3. 读取旧需求基线
4. 明确本次变更点
5. 更新 `00-需求总览.md`
6. 更新 `*-需求文档.md`
7. 更新 `*-技术方案.md`
8. 若影响页面则更新 `*-UI交互设计规范.md`
9. 更新任务拆解与测试用例
10. 更新实现控制总表 / 页面接口验收总表
11. 开发与开发自测
12. 覆盖率、代码审查、冒烟、QA、UI验收、产品验收
13. 如需要发布，补 `*-发布记录.md`

## 变更型额外要求

- 必须写清旧逻辑、本次变更点、回归范围
- 不允许只改代码不补差异说明
- 若变更影响公共接口、枚举、编码规则，必须回写到技术方案和公共规则引用处
- 变更发生在已归档任务时，先恢复对应任务再继续
- L2 / L3 必须先输出影响分析，再进入编码
- `doctor` 未通过前，不进入真实开发动作
