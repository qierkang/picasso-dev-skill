# 发布流转

## 前置条件

以下条件未全部满足前，禁止进入发布：

1. 开发自测通过
2. 代码审查无阻塞问题
3. 冒烟测试达到 `100% 通过`
4. QA / 产品验收通过

## 六步法

```text
前置卡点确认
  ↓
部署前检查
  ↓
执行发布
  ↓
发布后验证
  ↓
冒烟复核
  ↓
通知下游并回写发布记录
```

## 部署前检查

至少确认：

- 待发布 commit 明确
- 工作区无未提交脏变更
- 后端编译通过
- 前端构建通过
- 目标环境、配置变更、数据库变更已确认
- 回滚方式明确

## 执行发布

根据影响范围选择策略：

- 常规迭代：滚动更新
- 涉及 schema 变更：蓝绿或分阶段发布
- 涉及核心流程：灰度发布

## 发布后验证

至少验证：

- 服务健康检查
- 核心登录 / 权限接口
- 页面可访问
- 关键日志无 ERROR / FATAL

## 回滚原则

出现以下任一情况，立即评估回滚：

- 发布后验证失败
- 冒烟测试存在 FAIL
- QA 发现 `P0 / P1`
- 用户反馈核心链路异常

回滚后必须：

- 记录回滚原因
- 记录回滚目标版本
- 通知相关角色
- 确认服务恢复

## 发布记录

发布完成后，必须补齐：

- `*-发布记录.md`
- `stage-status.json`

必要时再把需求目录进入 `archives/`
