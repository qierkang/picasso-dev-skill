# 新需求工作流

## 目标

把一个新需求从原始输入推进到可开发、可测试、可验收状态。

## 标准阶段

1. 判断当前任务能力：`docs / dev / db / deploy`
2. 进入编码、SQL、启动前先运行对应 `doctor.sh`
3. 创建需求目录
4. 归档原型 / 截图 / 参考资料
5. 产出 `00-需求总览.md`
6. 产出 `*-需求文档.md`
7. 产出 `*-任务分解.md`
8. 产出 `*-技术方案.md`
9. 产出 `*-UI交互设计规范.md`
10. 产出 `*-测试用例.md`
11. 产出 `*-实现控制总表.md`
12. 产出 `*-页面接口验收总表.md`
13. 编码与开发自测
14. 覆盖率与关键流程覆盖校验
15. 代码审查
16. 两关制冒烟测试
17. QA 验证
18. UI 验收
19. 产品验收
20. 发布准备、发布验证与发布记录

## 环境前置规则

- 纯文档阶段可只通过 `bash install/doctor.sh --capability docs`
- 一旦进入真实编码，必须先通过 `bash install/doctor.sh --capability dev`
- 涉及本地 SQL / 数据回看，必须先通过 `bash install/doctor.sh --capability db`
- 涉及本地启动、冒烟、发布准备，必须先通过 `bash install/doctor.sh --capability deploy`
- `doctor` 失败时，先补环境，不继续写代码
- `deploy doctor` 除了工具链，还必须校验运行编排入口：后端启动命令、前端启动命令、就绪地址、自动登录配置
- `doctor` 通过后，默认直接继续执行本地启动、联调、冒烟、收尾，不反复询问是否继续

## 强制 Gate 顺序

```text
需求文档
  ↓
任务分解卡点
  ↓
技术方案 / UI 规范
  ↓
编码开发
  ↓
开发自测卡点
  ↓
覆盖率卡点
  ↓
代码审查卡点
  ↓
冒烟测试卡点
  ↓
QA 验证卡点
  ↓
UI 验收卡点
  ↓
产品验收卡点
  ↓
发布准备 / 发布验证 / 回滚预案
```

## 默认放行结论

- 每个阶段默认结论都是 `NEEDS_WORK`
- 没有证据的“通过”默认无效
- 同一任务连续失败 `3` 次，必须升级给 PM / 架构重拆
- 冒烟阶段默认由 `stage-gate.py smoke` 真实执行脚本，自动启动前后端并回收本地进程
