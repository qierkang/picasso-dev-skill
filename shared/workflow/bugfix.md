# Bug 修复工作流

## 目标

复现问题、定位根因、完成修复并控制回归风险。

## 标准阶段

1. 判断当前任务能力：`docs / dev / db / deploy`
2. 进入复现、SQL、启动前先运行对应 `doctor.sh`
3. 记录问题现象与复现路径
4. 确认影响范围
5. 产出修复方案
6. 更新任务拆解与测试用例
7. 实施修复
8. 开发自测与回归验证
9. 代码审查与必要冒烟
10. QA 验证
11. UI / 产品确认（如涉及页面或业务口径）
12. 如已进入提测或上线窗口，补发布记录

## 最低要求

- 不跳过复现记录
- 不跳过影响范围说明
- 不只修症状，不记录根因
- 不允许跳过开发放行报告和代码审查报告
- `doctor` 未通过前，不继续真实复现和修复
