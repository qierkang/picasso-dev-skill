# 服务端需求变更流程

当任务已进入 `active/`，并且用户对服务端需求提出调整时，执行这份规则。

## 变更分级

| 级别 | 定义 | 影响范围 | 处理方式 |
|------|------|----------|----------|
| `L1` 轻微 | 文案修正、别名调整、非核心说明补充 | 不影响接口、数据结构、核心逻辑 | 记录变更日志后继续 |
| `L2` 一般 | 新增或删除非核心字段、校验规则调整、枚举值变更 | 影响 specs / design / tasks 之一 | 回退受影响文档到待确认，再继续 |
| `L3` 重大 | 数据模型调整、核心流程变更、新增模块 | 影响多个文档或已完成代码 | 先做影响分析，再决定拆新任务或整体回退 |

## 标准流程

1. 记录用户的变更内容
2. 判断变更级别 `L1 / L2 / L3`
3. 输出影响分析
4. 等用户确认处理方案
5. 更新 `design.md / tasks.md / status.md`
6. 再组织编码模型执行修改
7. 回填变更记录

## 影响分析至少包含

- 变更描述
- 影响级别
- 受影响文档
- 受影响代码任务
- 是否需要补 `MENU_DATA.md / DICT_DATA.md / API_DOC.md`
- 是否需要恢复已归档任务

## 变更执行约束

- 未经确认，不直接改代码
- 先改 `design.md`，再改 `tasks.md`
- 变更任务必须追加到 `tasks.md` 中，不要只口头说明
- 已归档任务发生变更时，优先恢复到 `active/` 再继续

## 变更记录格式

追加到 `status.md`：

```markdown
### 变更 #N — YYYY-MM-DD
- 变更描述：
- 影响级别：L1 / L2 / L3
- 受影响文档：
- 受影响代码：
- 处理方式：
- 状态：进行中 / 已完成
```
