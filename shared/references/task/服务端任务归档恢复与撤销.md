# 服务端任务归档恢复与撤销

这份规则用于管理服务端任务从 `active` 到 `archived` 的流转，以及后续恢复、补产物、撤销。

## 归档任务

满足以下条件后，才允许进入归档：

1. `status.md` 已更新到最新
2. `tasks.md` 最终任务已完成
3. 联调或接口交接资料已确认
4. `openspec validate --change <任务名>` 通过

标准流程：

1. 提示用户是否归档
2. 运行 `openspec validate --change <任务名>`
3. 检查 `MENU_DATA.md / DICT_DATA.md` 是否齐全
4. 执行 `openspec archive <任务名> --yes`
5. 将 `${PICASSO_SERVER_TASK_DIR}/active/<任务名>/` 移动到 `archived/`
6. 更新 `status.md` 当前阶段为“已归档”
7. 视需要复制设计产物和 SQL 到归档目录

## 补复制产物

归档后，建议把以下文件复制到 `${PICASSO_SERVER_TASK_DIR}/archived/<任务名>/`：

- `MENU_DATA.md`
- `DICT_DATA.md`
- `API_DOC.md`（如存在）
- `tasks.md` 中引用的 SQL 脚本

复制前先确认源文件真实存在，不要伪造缺失产物。

## 恢复已归档任务

当已归档任务需要继续开发或处理变更时：

1. 确认 `${PICASSO_SERVER_TASK_DIR}/active/<任务名>/` 不存在
2. 将 `${PICASSO_SERVER_TASK_DIR}/archived/<任务名>/` 移回 `active/`
3. 如对应 `openspec` 变更已进入归档目录，同时恢复到 `openspec/changes/<任务名>/`
4. 按 `服务端需求变更流程.md` 继续推进

## 撤销任务

仅当用户明确要求“撤销任务 / 删除任务 / 当作没发生过”时执行。

执行原则：

- 先说明这是不可逆操作
- 已 push 的远程代码，不自动强推回退
- 只清理当前任务相关内容，不误伤其他任务

推荐顺序：

1. 清理 `openspec/changes/<任务名>` 或 `openspec/changes/archive/<任务名>`
2. 删除 `incoming / active / archived` 中对应任务目录
3. 未提交代码按安全方式回退
4. 任务分支仅在用户确认后删除
5. 同步清理与该任务有关的记忆和说明
