# 两关制冒烟测试说明

## 目标

避免只做接口验证、不做页面集成检查，导致大量集成问题漏到用户手上。

## 关卡 1：API 接口验证

至少覆盖：

- 服务健康检查
- 登录获取 Token
- 核心 CRUD 接口
- 字典 / 菜单 / 权限依赖

## 关卡 2：页面集成验证

至少覆盖：

- PC 页面文件与路由
- API 层文件与请求路径
- 小程序 / App 路由注册（如涉及）
- 后端 Controller 路径与前缀
- 编译通过

## 高频遗漏项

旧项目里最容易漏的不是 CRUD 本身，而是这些集成项：

- 字典类型已设计但未真正创建
- Controller 放错目录，导致 `/admin-api` 或 `/app-api` 前缀错误
- 小程序 `pages.json` 或 TabBar 未注册
- Flutter 路由名、页面映射、`Navigation.to` 分支遗漏
- 隧道、代理、远程 API 地址未联通

冒烟脚本至少要把本次需求涉及的高频遗漏项显式检查掉，不能只留在人工记忆里。

## 通过标准

- 两关都通过，且总结果达到 `100% 通过`
- 任一 FAIL 都必须回流修复
- 冒烟报告必须给出第 1 关和第 2 关各自的通过 / 失败汇总
