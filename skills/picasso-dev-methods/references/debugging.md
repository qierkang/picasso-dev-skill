# Systematic Debugging

## 目标

把排障从“猜原因”改成“先复现、再定位、后修复、最后回归”。

## 标准步骤

1. 明确现象
2. 固化复现路径
3. 确认影响范围
4. 缩小可疑范围
5. 收集日志、输入、状态、数据差异
6. 证明根因，而不是只找到相关代码
7. 修复后做回归验证

## 最低证据

- 复现条件
- 根因说明
- 修复点
- 回归结果

## Picasso 落地要求

Bug 修复至少要同步更新：

- `*-需求文档.md` 或差异说明
- `*-技术方案.md`
- `*-开发放行报告.md`
- 必要时补 `*-测试用例.md`

## 常见反模式

- 直接改代码，不先确认是否能稳定复现
- 看到异常栈就立刻下结论
- 修好了主路径，但没有验证边界与回归影响
