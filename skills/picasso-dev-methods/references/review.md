# Review Handling

## 目标

把代码审查从“礼貌性接受”变成“基于事实判断是否采纳”。

## 默认做法

1. 先确认意见是否准确描述了问题
2. 再确认是否真会造成行为风险、维护成本或质量问题
3. 区分：
   - 必改
   - 可优化
   - 不采纳
4. 给出理由，并在文档 / 代码 / 测试中留痕

## 审查优先级

1. 正确性和数据风险
2. 边界行为和回归风险
3. 可维护性
4. 风格一致性

## Picasso 落地要求

- 审查结果回填到 `*-代码审查报告.md`
- 阻塞级问题未关闭，不得进入下一阶段
- 如不采纳某条意见，需要写明判断依据和影响

## 常见误区

- 不验证就直接照改
- 把风格建议当成阻塞问题
- 只回复“已处理”，不留下证据
