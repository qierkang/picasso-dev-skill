# Planning

## 目标

在进入编码前，先把需求收敛成可执行路径，避免直接开写导致返工。

## 默认做法

1. 先判断任务类型：`new-feature / change-request / bugfix / ui-optimization`
2. 明确平台范围：`backend / pc / miniapp / mobile_app / config`
3. 生成或更新 `manifest.json`
4. 将工作拆进对应主产物：
   - `*-需求文档.md`
   - `*-技术方案.md`
   - `*-任务分解.md`
   - `*-实现控制总表.md`
5. 把任务拆到 `30-60` 分钟粒度，并明确输入、输出、验收标准、证据

## 规划检查点

- 范围是否闭合
- 平台边界是否清晰
- 是否存在配置、菜单、字典、SQL 依赖
- 是否需要 UI 设计或服务端协同
- 是否需要本地启动、联调、冒烟
- 是否已经识别风险、假设和非本次范围

## 失败信号

- 需求名很大，任务粒度很粗
- 文档里只有“做什么”，没有“怎么验收”
- 没有把边界条件、异常状态、权限差异写清楚
- 没有在进入编码前完成主产物链路

## 在 Picasso 中的落地要求

规划结果必须回填到 `workspace/requests/<request-key>/`，不能只停留在对话里。
