# TDD

## 目标

把“先写功能，再补测试”收紧为“先明确验证，再实现”，降低回归风险。

## 默认做法

1. 先定义要验证的业务行为、边界和失败场景
2. 优先补单测 / 集成测试 / 页面关键流程清单
3. 再做最小实现通过验证
4. 通过后再做必要重构

## Picasso 中的适用范围

- 后端接口、服务、领域规则
- 前端关键交互流程
- Bug 修复中的复现用例
- 配置或脚本类能力的回归校验

## 最低要求

- 后端：补充或更新自动化测试，覆盖新增规则和回归风险
- 前端：更新关键流程覆盖清单，必要时补自动化或至少补手工验证步骤
- Bug：先固化复现条件，再修复

## 不适用的误区

- 纯文档任务不需要强行写测试代码
- 无法自动化的 UI 细节，也至少要写成明确验收步骤
- 不允许用“以后再补测试”替代当前质量闭环
