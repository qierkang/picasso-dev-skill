# DEC-0004 分层设计底座与平台 UI 专项层

## 状态

已采纳

## 日期

2026-04-12

## 背景

`picasso-dev-skill` 原先已经具备完整的需求、方案、开发、验收和发布流程，但 UI 能力主要集中在：

- 产出 `UI交互设计规范.md`
- 对齐三端差异
- 用 UI 验收报告收口

这套做法能保证“流程有壳”，但无法保证“页面真的设计得更好”。

用户已经明确反馈：

- PC、HTML、iOS、Android、小程序的页面设计交互偏弱
- 希望吸收外部设计 skill 的优势
- 但不希望直接把外部大包整包塞进主入口

## 决策

1. 设计能力采用“三层收敛”：
   - 通用入口层：`picasso-dev-ui`
   - 设计底座层：`picasso-dev-design-system`
   - 平台专项层：`picasso-dev-ui-web / ios / android / miniapp`
2. 通用设计知识统一放入 `shared/references/design/`
3. UI 审查单独由 `picasso-dev-ui-review` 承接
4. 外部 skill 只吸收方法，不镜像整包作为运行依赖

## 原因

- 保持主入口小而清晰
- 让设计知识能按平台按需加载
- 避免 `picasso-dev-ui` 演化成难维护的大杂烩
- 保留 Picasso 原有流程优势，同时补齐设计能力短板

## 影响

正向影响：

- 新需求进入 UI 阶段时，设计意图更完整
- 多端页面的交互差异有显式依据
- UI 验收可以对准 token、动效、可访问性

成本：

- 需要维护更多 companion skill
- 文档与模板结构会比过去更细

## 约束

1. 设计底座层只做高价值共性规则，不堆砌百科式内容
2. 平台专项层必须按需加载，不能默认全量读取
3. 新增平台专项前，先判断是否真有长期复用价值
