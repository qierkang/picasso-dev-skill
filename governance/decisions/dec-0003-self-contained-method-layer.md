# DEC-0003 自包含方法层与显式适配层

## 状态

已采纳

## 日期

2026-04-09

## 背景

`picasso-dev-skill` 已具备主流程、专项 skill、模板与治理能力，但此前缺少显式的方法层与运行端适配层。

这会带来两个问题：

1. 外部高价值 skill 方法难以稳定吸收，只能口头借鉴
2. Claude、Codex、OpenCode、OpenClaw 等不同运行端的接入关系不够直观

同时，用户已明确要求：

- 技能包应作为可迁移的一体包分发
- 禁止依赖宿主机外部 skill 路径
- 可以在包内预留 vendor 空间，但主流程不能被外部框架接管

## 决策

1. 新增 `skills/picasso-dev-methods/` 作为方法层
2. 新增 `vendor/` 作为外部内容镜像预留区
3. 新增 `governance/vendor-skills.yaml` 管理外部来源与采用策略
4. 在仓库根目录增加显式适配层目录：
   - `.claude-plugin/`
   - `.codex/`
   - `.opencode/`
   - `.openclaw/`
5. 适配层仅承担接入说明和平台边界说明，不复制主流程

## 原因

- 提升跨机器迁移的稳定性
- 降低新人理解目录结构的成本
- 允许持续吸收外部方法，但避免运行时外部依赖
- 保持 `picasso-dev` 作为唯一主入口，不引入第二套总控框架

## 影响

正向影响：

- 方法能力具备明确归属
- 运行端适配关系更清晰
- 后续对照外部文章和 skill 时，有稳定承接位置

成本：

- 目录层级略有增加
- 维护时需要同时更新方法层、适配层和治理记录

## 后续约束

1. 如需吸收新的外部 skill，先登记 `vendor-skills.yaml`
2. 只有当必须保留完整副本时，才写入 `vendor/skills/`
3. 不允许在任何主流程说明中把宿主机外部 skill 写成必需前置条件
