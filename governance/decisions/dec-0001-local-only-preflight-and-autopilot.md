# dec-0001 local-only 环境预检与自主执行

## 状态

accepted

## 日期

2026-04-09

## 决策

`picasso-dev-skill` 自本次起固定采用以下默认规则：

1. 所有 Picasso 开发线程先判断当前能力属于 `docs / dev / db / deploy`
2. 进入真实开发前必须先通过对应能力的 `doctor.sh`
3. skill 只允许 `local / dev` 作为可操作环境边界
4. `test / uat / prod` 只用于识别边界，不允许连接、查询、部署
5. 环境通过后，routine 动作默认自主推进，不反复向使用者确认

## 背景

- 该 skill 将被复制到新机器、虚拟机和不同模型环境中使用
- 用户明确要求新环境接入时先自检，再安装缺失环境，直到可开发为止
- 用户明确要求数据库查询和部署只能在本地边界内完成，不能误连测试、UAT、生产
- 用户更关注“功能开发完成 + 访问地址 + 账号密码 + 验证路径”的结果，不希望过程里持续被询问是否继续

## 影响

- `docs` 场景只校验最小运行条件，不再被完整开发工具链阻塞
- `dev / deploy` 场景必须校验 Java / Maven / Node / pnpm / Flutter 及 Picasso 仓库要求
- `db` 场景必须校验本地数据库配置、host 边界和 `psql`
- prompt、模板、manifest 和 workflow 必须显式携带环境边界与 `doctor` 命令

## 代价与权衡

- 新机器首次进入开发前会多一次显式体检，启动成本略高
- 但能显著降低“写到一半才发现环境错了”以及“误碰 test / uat / prod”的风险
- 自主推进会减少过程确认次数，但要求 skill 对阻塞分类更清晰
