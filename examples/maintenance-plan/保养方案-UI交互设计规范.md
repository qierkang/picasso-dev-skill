# 保养方案 UI 交互设计规范

## 元信息

- 版本：`v1.0`
- 日期：`2026-04-12`
- 编写人：`qierkang+codex`
- 状态：`草稿`

## 1. 全局规范

- 目标用户：设备管理员、设备工程师、系统管理员
- 业务目标：高效维护保养方案与保养项明细，保证主数据一致性与删除安全
- 终端范围：`PC Web 首发，Android / 微信小程序预留`
- 页面类型：`列表页 + 详情页`

## 2. 页面定位

- 主任务：维护保养方案主数据和保养项明细
- 高频操作：查询、添加、编辑、查看、删除
- 关键风险点：
  - 系统字段来源不准导致公司信息错误
  - 保养项自动带出字段被误做成可编辑
  - 查看态与编辑态按钮行为不一致
  - 删除动作缺失引用校验提示

## 3. 设计策略与风格方向

- 风格方向：`高密度运营台 + 流程型表单台`
- 页面气质关键词：`稳重、清晰、强规则、低装饰`
- 品牌 / 业务调性说明：设备与保养主数据场景优先强调规则准确性和只读字段可信度
- 信息层级原则：列表先筛选与状态，详情页先主表，再明细
- 危险操作与主操作的优先级原则：`提交` 主、`删除` 危险、`取消` 次要

## 4. 三端范围矩阵

| 模块/页面 | PC Web | Android App | 微信小程序 | 说明 |
|-----------|--------|-------------|------------|------|
| 保养方案列表 | 本期实现 | 预留 | 预留 | 先以 PC 为准完成页面结构 |
| 保养方案详情 | 本期实现 | 预留 | 预留 | 后续端侧复用接口和字段规则 |
| 保养项明细维护 | 本期实现 | 预留 | 预留 | 保持主表 + 子表同口径 |

## 5. 全局展示规范

- 编码与名称展示格式：
  - `设备类型`：优先显示 `编码 + 名称`
  - `保养项`：优先显示 `编码 + 名称`
- 系统字段默认展示格式：
  - `公司编码` 只读文本
  - `公司名称` 只读文本
- 列表与详情的显示差异：
  - 列表页强调可筛选与状态标签
  - 详情页强调字段来源、只读字段和明细回填
- 只读字段表现形式：
  - 统一浅灰背景输入框或只读文本块
  - 不允许用普通可编辑输入框伪装只读

## 6. 设计 Token 与组件策略

- 主色 / 辅色方向：主色稳重，状态标签以成功 / 警告 / 风险三类清晰区分
- 字体 / 字重方向：标题 `semibold`，正文 `regular`，只读字段与状态标签用 `medium`
- 圆角 / 阴影方向：采用轻圆角和轻阴影，不做营销化玻璃效果
- 列表页组件策略：优先表格和状态标签
- 详情页组件策略：主表表单 + 明细表格
- 表单控件策略：只读字段统一浅灰只读样式

## 7. 页面结构与操作路径

### 7.1 页面分区

- 头部区：标题、返回、页面操作说明
- 搜索筛选区：保养方案编码、保养方案名称、适用设备类型、单据类型
- 数据展示区：列表表格
- 操作区：添加、查询、重置；行内编辑、删除、查看
- 明细区：详情页中的保养项明细子表

### 7.2 核心交互路径

| 场景 | 操作步骤 | 预期结果 |
|------|----------|----------|
| 查询列表 | 输入筛选条件 -> 点击查询 | 列表按条件过滤，默认保持创建时间倒序 |
| 新增方案 | 点击添加 -> 打开详情页 -> 填写方案名称、设备类型、状态 -> 选择保养项 -> 提交 | 保存成功并返回列表 |
| 编辑方案 | 点击编辑 -> 回显数据 -> 修改主表或明细 -> 提交 | 保存成功，列表刷新 |
| 查看方案 | 点击查看 -> 打开只读详情页 | 所有字段只读，保养项明细可查看不可编辑 |
| 删除方案 | 点击删除 -> 弹窗确认 -> 后端引用校验 | 通过则删除，不通过则提示被引用原因 |

## 8. 系统字段交互规格

| 字段 | PC 组件形态 | Android 组件形态 | 小程序组件形态 | 是否弹窗/独立页 | 搜索区 | 单选/多选 | 回填字段 | 展示格式 | 默认值来源 | 是否允许改选 |
|------|-------------|------------------|----------------|-----------------|-------|-----------|---------|---------|-----------|-------------|
| 公司编码 | 只读输入框 | 只读 Cell | 只读 Cell | 否 | 否 | / | `companyCode/companyName` | 公司编码 | 当前登录公司 | 否 |
| 设备类型编码 | Select / 选择器 | 选择页 | 选择页 | 可选独立页 | 是 | 单选 | `deviceTypeCode/deviceTypeName` | 编码-名称 | 无 | 是 |
| 保养项编码 | Select / 选择器 | 选择页 | 选择页 | 可选独立页 | 是 | 单选 | `itemCode/itemName/itemMethod/itemContent/itemStatus` | 编码-名称 | 无 | 是 |

## 9. 弹窗 / 选择器 / 独立页规范

- 哪些字段使用选择器：
  - `设备类型编码`
  - `保养项编码`
- 哪些字段使用独立选择页：
  - 移动端建议独立选择页
- 关闭后是否保留草稿态：
  - 详情页取消时若有改动，需二次确认
- 是否支持二次改选：
  - 设备类型允许改选
  - 保养项允许改选，但改选后必须整行重刷带出字段

## 10. 状态反馈与动效规范

- 页面切换动效：轻过渡，避免夸张动画影响表单识读
- 弹窗 / 抽屉开合反馈：如后续做选择器弹层，控制在 `180-220ms`
- 提交 / 保存 / 删除反馈：loading + 成功 / 失败提示
- 加载与骨架屏策略：详情加载和子表加载可分开骨架
- 结果页或成功态强化策略：本场景不需要情绪化强化，优先规则清晰

## 11. 关闭与保存交互规范

- 保存成功：
  - 返回列表页
  - 列表刷新并定位到最新数据
- 保存失败：
  - 停留当前页
  - 提示明确失败原因
- 关闭弹窗：
  - 本模块主交互建议使用独立页，不建议主表编辑放弹窗
- 有未保存修改时：
  - 点击取消需弹确认框

## 12. 表单校验与联动

| 字段 | 校验规则 | 联动触发条件 | 联动结果 | 备注 |
|------|---------|-------------|---------|------|
| 保养方案名称 | 必填，建议 <= 100 字符 | 提交前 | 阻断提交 | - |
| 设备类型编码 | 必选 | 选择设备类型后 | 自动带出设备类型名称 | 名称只读 |
| 状态 | 必选 | 页面初始化 | 默认启用 | 最终状态集待确认 |
| 保养项编码 | 必选 | 选择保养项后 | 自动带出名称/方法/内容/状态 | 带出字段只读 |
| 保养项内容 | 不允许空 | 保养项带出后 | 展示长文本 | 如基础数据为空则阻断提交 |

## 13. 状态与按钮显隐

| 场景 | 页面状态 | 按钮集合 | 是否可编辑 | 备注 |
|------|----------|---------|-----------|------|
| 列表页 | 默认 | 添加 / 查询 / 重置 / 编辑 / 删除 / 查看 | 是 | 行内操作固定右侧 |
| 详情页新增态 | 新增 | 取消 / 提交 | 是 | 公司字段与编码字段只读 |
| 详情页编辑态 | 编辑 | 取消 / 提交 | 是 | 明细可调整 |
| 详情页查看态 | 查看 | 编辑 / 关闭 | 否 | 点击编辑后切换为编辑态 |

## 14. 空态 / 加载态 / 错误态

| 场景 | PC | Android | 小程序 | 说明 |
|------|----|-----------|---------|------|
| 列表无数据 | 展示空态插画 + 返回查询建议 | 预留 | 预留 | 明确“暂无保养方案” |
| 详情加载中 | 骨架屏 / loading | 预留 | 预留 | 明细区单独 loading |
| 删除校验失败 | 弹窗提示被引用原因 | 预留 | 预留 | 不允许只提示“删除失败” |

## 15. 可访问性与响应式要求

- PC / Web 断点策略：优先桌面表格和双列表单，窄屏下允许主表折为单列
- 移动端热区要求：预留端主按钮和行操作热区不小于 `44x44`
- 键盘 / 焦点 / 读屏要求：PC 焦点态清晰，状态字段不能只靠颜色表达
- 文本放大与超长文案处理：长文本保养内容允许换行，不允许遮挡操作按钮
- 错误提示定位策略：错误就近展示到字段或明细行

## 16. 视觉与端差异说明

- PC：
  - 主表使用双列或三列整齐表单布局
  - 明细区建议表格形式，不要退化成口语化块状文本
- Android：
  - 后续建议分段卡片式表单
- 小程序：
  - 后续建议分段 Cell + 明细折叠区
- App / H5（如涉及）：
  - 当前不在本轮落地范围内
- 危险操作与主操作的视觉层级：
  - `提交` 作为主按钮
  - `删除` 在列表中使用危险色文案或图标

## 17. 特殊说明

- 是否依赖内部公共接口回填：是，依赖公司、设备类型、保养项、编码规则
- 是否存在系统字段只读但必须展示：是，公司编码、公司名称、保养方案编码、保养项带出字段
- 是否存在审批状态 / 编码 / 字典值特殊展示：是，保养方案编码必须走内部编码规则

## 18. 开发与验收约束

- 前端 / 小程序 / App 实现时必须回对的组件语义：只读字段、状态标签、危险操作样式必须一致
- 哪些差异允许按端做本地优化：预留端可把选择器升级为独立页
- 哪些交互不能随意改：保养项带出字段只读、删除引用校验、查看态不可编辑

## 19. 验收要点

- 查看态所有带出字段必须只读。
- 选择保养项编码后，名称、方法、内容、状态必须一次性自动带出。
- 删除失败时必须展示“被引用”的明确提示，而不是统一失败提示。
- 列表默认排序必须按创建时间倒序。
