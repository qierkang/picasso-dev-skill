## 1. 技术架构

- 架构目标：模块边界清晰、主子数据结构稳定、删除校验可追踪、公共接口依赖可替换。
- 技术形态：`后端新增模块接口 + PC 前端新增列表/详情页面`。
- 依赖系统：
  - 登录上下文公司信息
  - 组织/部门管理能力
  - 设备类型基础数据
  - 保养项基础数据
  - 动态编码规则服务
  - 引用校验服务
- 隔离策略：
  - 保养方案模块使用独立菜单、独立接口、独立表结构。
  - 公共主数据通过复用接口获取，不复制基础数据逻辑。

## 2. 范围说明

- 本方案覆盖：
  - 保养方案列表查询
  - 保养方案主表维护
  - 保养项明细维护
  - 删除前引用校验
  - 后端接口与 PC 页面联调
- 本方案不覆盖：
  - 审批流接入
  - 导入导出
  - 小程序 / App 页面开发
  - 删除引用来源模块的新增开发

## 3. 数据表设计

### 3.1 数据表数量口径

- 新增业务表：`2 张`
- 复用基础表：公司/组织表、设备类型表、保养项主数据表、用户租户表
- 删除 / 校验依赖表：引用校验来源表待确认

| 表名 | 类型 | 用途 | 核心字段 | 备注 |
|------|------|------|---------|------|
| `maintenance_plan` | 主表 | 存储保养方案主信息 | company_code/plan_code/plan_name/device_type_code/status | 新增 |
| `maintenance_plan_item` | 子表 | 存储保养方案保养项明细 | plan_id/item_code/item_name/item_method/item_content/item_status | 新增 |

### 3.2 主表设计

- 表名：`maintenance_plan`
- 核心字段：`company_code`、`company_name`、`plan_code`、`plan_name`、`device_type_code`、`device_type_name`、`status`
- 主键策略：`id BIGINT`
- 基础字段继承方式：统一使用框架基础字段 `creator/create_time/updater/update_time/deleted/tenant_id`
- 删除策略：逻辑删除；删除前必须先做引用校验

```sql
CREATE TABLE maintenance_plan (
    id                BIGSERIAL PRIMARY KEY,
    company_code      VARCHAR(64)  NOT NULL,
    company_name      VARCHAR(128) NOT NULL,
    plan_code         VARCHAR(64)  NOT NULL,
    plan_name         VARCHAR(200) NOT NULL,
    device_type_code  VARCHAR(64)  NOT NULL,
    device_type_name  VARCHAR(200) NOT NULL,
    status            SMALLINT     NOT NULL DEFAULT 1,
    creator           VARCHAR(64)  DEFAULT '',
    create_time       TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updater           VARCHAR(64)  DEFAULT '',
    update_time       TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted           SMALLINT     NOT NULL DEFAULT 0,
    tenant_id         BIGINT       NOT NULL DEFAULT 0
);

ALTER TABLE maintenance_plan
    ADD CONSTRAINT uk_maintenance_plan_code UNIQUE (plan_code, tenant_id, deleted);
```

### 3.3 子表 / 明细表设计

- 表名：`maintenance_plan_item`
- 关联字段：`plan_id -> maintenance_plan.id`
- 冗余字段：保养项名称、保养方法、保养内容、状态
- 更新策略：编辑主表时采用“先校验、后全量替换子表”策略
- 主子级联规则：主表逻辑删除前必须先执行引用校验；删除通过后主子一起删除

```sql
CREATE TABLE maintenance_plan_item (
    id                BIGSERIAL PRIMARY KEY,
    plan_id           BIGINT       NOT NULL,
    item_code         VARCHAR(64)  NOT NULL,
    item_name         VARCHAR(200) NOT NULL,
    item_method       VARCHAR(200) NOT NULL,
    item_content      TEXT         NOT NULL,
    item_status       SMALLINT     NOT NULL,
    creator           VARCHAR(64)  DEFAULT '',
    create_time       TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updater           VARCHAR(64)  DEFAULT '',
    update_time       TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted           SMALLINT     NOT NULL DEFAULT 0,
    tenant_id         BIGINT       NOT NULL DEFAULT 0
);

CREATE INDEX idx_maintenance_plan_item_plan_id
    ON maintenance_plan_item (plan_id, deleted, tenant_id);
```

### 3.4 索引与约束建议

- 是否使用业务唯一校验：`plan_code` 唯一；同一方案下 `item_code` 不允许重复
- 是否涉及引用校验：是，删除前必须先查外部业务引用
- rollback 策略：
  - 主表、子表同事务提交
  - 删除校验失败则整单回滚

## 4. 数据与接口约束

### 4.1 前端提交字段

- 必须由前端显式传值的字段：
  - `planName`
  - `deviceTypeCode`
  - `status`
  - `items[].itemCode`
- 前端只读展示但允许透传的字段：
  - `deviceTypeName`
  - `items[].itemName`
  - `items[].itemMethod`
  - `items[].itemContent`
  - `items[].itemStatus`

### 4.2 后端自动补齐字段

- 登录人 / 租户 / 公司 / 部门相关字段：
  - `companyCode` 根据当前登录公司补齐
  - `companyName` 根据 `companyCode` 查询公共组织/公司能力补齐
  - `tenantId` 框架自动补齐
- 系统生成编码字段：
  - `planCode` 通过动态编码规则服务生成
- 状态默认值与流转补齐字段：
  - `status` 默认 `启用`，具体枚举值由字典或枚举定义确定

### 4.3 系统字段交互技术说明

| 字段 | 组件类型 | 是否弹窗 | 搜索区 | 单选/多选 | 回填字段 | 前端传值 | 后端补齐 | 备注 |
|------|---------|---------|-------|-----------|---------|---------|---------|------|
| `companyCode` | 只读输入框 | 否 | 否 | / | `companyCode/companyName` | 否 | 是 | 来自登录上下文 |
| `deviceTypeCode` | 下拉 / 选择器 | 可选 | 是 | 单选 | `deviceTypeCode/deviceTypeName` | 是 | 可校验 | 设备类型名称可由后端二次校验补齐 |
| `itemCode` | 下拉 / 选择器 | 可选 | 是 | 单选 | `itemCode/itemName/itemMethod/itemContent/itemStatus` | 是 | 是 | 仅允许选择启用保养项 |

### 4.4 日期 / 格式化字段技术说明

- 日期格式：
  - 列表创建时间展示为 `yyyy-MM-dd HH:mm:ss`
- 展示拼接字段：
  - 本模块暂无强制拼接显示字段
- 冗余展示字段：
  - `companyName`
  - `deviceTypeName`
  - `itemName`
  - `itemMethod`
  - `itemContent`
  - `itemStatus`

## 5. 接口设计

### 5.1 接口数量口径

- 新增业务接口：`6 个`
- 复用基础接口：`4 个`
- 合计涉及接口能力：`10 个`

### 5.2 新增业务接口清单

| 接口 | 方法 | 说明 | 参数 | 返回值 | 权限标识 | 接口端 |
|------|------|------|------|--------|---------|-------|
| `/maintenance/plan/page` | GET | 保养方案分页查询 | pageNo/pageSize/planCode/planName/deviceTypeCode/billType | `PageResult<MaintenancePlanRespVO>` | `maintenance:plan:query` | 后端/PC |
| `/maintenance/plan/get` | GET | 保养方案详情 | id | `MaintenancePlanDetailRespVO` | `maintenance:plan:query` | 后端/PC |
| `/maintenance/plan/create` | POST | 新增保养方案 | `MaintenancePlanCreateReqVO` | `Long` | `maintenance:plan:create` | 后端/PC |
| `/maintenance/plan/update` | PUT | 编辑保养方案 | `MaintenancePlanUpdateReqVO` | `Boolean` | `maintenance:plan:update` | 后端/PC |
| `/maintenance/plan/delete` | DELETE | 删除保养方案 | id | `Boolean` | `maintenance:plan:delete` | 后端/PC |
| `/maintenance/plan/check-delete` | GET | 删除前引用校验 | id | `DeleteCheckRespVO` | `maintenance:plan:delete` | 后端/PC |

### 5.3 复用接口清单

| 接口能力 | 方法 | 接口路径 | 复用原因 | 备注 |
|---------|------|---------|---------|------|
| 设备类型列表 | GET | `/equipment/type/simple-list` | 复用系统设备类型主数据 | 选择设备类型编码与名称 |
| 保养项列表 | GET | `/maintenance/item/simple-list` | 复用保养项基础数据 | 选择保养项后自动回填明细 |
| 动态编码规则 | POST/GET | 内部编码规则接口 | 统一生成保养方案编码 | 不允许前端本地拼编码 |
| 组织/公司信息查询 | GET | 内部组织/公司接口 | 根据公司编码带出公司名称 | 来源模块需与现网一致 |

### 5.4 内部公共接口 / 基础能力依赖

| 依赖类型 | 来源模块 / 接口 | 用途 | 获取时机 | 失败处理 | 备注 |
|---------|----------------|------|---------|---------|------|
| 登录上下文 | 框架登录态 | 获取当前公司编码、租户信息 | 打开新增页 / 保存前 | 无登录态则禁止进入 | 系统级必需 |
| 组织/公司能力 | 内部组织/公司接口 | 根据公司编码带出公司名称 | 新增初始化 / 保存校验 / 详情回显 | 返回明确错误并阻断保存 | 原型说明来自部门管理 |
| 设备类型基础数据 | 设备类型接口 | 设备类型编码选择与名称回填 | 新增/编辑页选择时 | 禁止保存并提示基础数据缺失 | 建议只返回启用数据 |
| 保养项基础数据 | 保养项接口 | 保养项编码选择与字段回填 | 新增/编辑明细选择时 | 阻断提交 | 查询条件必须带主表公司编码 |
| 动态编码规则 | 内部编码服务 | 生成保养方案编码 | 创建时 | 返回失败则整单创建失败 | 应用场景固定为保养方案 |
| 删除引用校验 | 来源模块待确认 | 删除前校验方案是否被引用 | 点击删除时 | 返回引用明细并阻断删除 | 必须先于逻辑删除执行 |

> 凡是系统已有能力，例如公司、字典、编码规则，都必须以接口或公共服务方式接入，不得重新本地实现一份。

## 10. 状态设计

- 无审批流时：
  - 默认新增为 `启用` 或系统默认值
  - 编辑不改变状态
  - 删除需先校验再逻辑删除
- 有审批流时：
  - 本期不接入，仅预留字段和状态扩展点
- 非法状态流转说明：
  - 被引用的方案不能删除
  - 查看态不能直接提交编辑结果

## 11. 验证要求

- 前端：
  - 列表查询、详情回显、只读态切换、保养项回填正确
- 后端：
  - 主子表事务一致
  - 删除校验先于删除执行
  - 编码规则失败时创建整体失败
- 浏览器：
  - 控制台无报错
  - Network 参数与返回值符合方案
- SQL / 字典 / 菜单：
  - 状态枚举、单据类型枚举（若保留）、菜单权限、设备类型和保养项基础数据均可用

## 12. 风险与残余项

- 删除引用校验真实来源模块未最终确认，真实开发前必须先补齐。
- 单据类型字段是否保留未确认，可能影响表结构和查询接口。
- 公司名称来源模块原型描述为“部门管理”，实际项目中可能落在公司/组织中心，需以现网能力为准。
