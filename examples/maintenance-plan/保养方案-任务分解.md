# 保养方案 任务分解

## 元信息

- 版本：`v1.0`
- 日期：`2026-04-08`
- 编写人：`qierkang+codex`
- 状态：`草稿`
- 默认放行立场：`NEEDS_WORK`
- 默认任务粒度：`30-60 分钟`

## 1. 任务总览

- 需求目标：完成保养方案模块的后端 + PC 首发版本
- 当前范围：
  - 列表页
  - 详情页
  - 主表 + 子表维护
  - 删除前引用校验
- 不在本轮范围：
  - 小程序 / App 页面开发
  - 导入导出
  - 审批流
- 总体完成标准：
  - 文档基线齐全
  - 主子表接口稳定
  - 页面满足原型主要交互
  - 删除校验逻辑可靠

## 1.1 任务切片规则

- 每个开发任务尽量收敛到 `30-60` 分钟可独立推进
- 每个任务必须能映射到 `实现控制总表` 或 `页面接口验收总表`
- 先公共依赖确认，再进入真实编码
- 未明确来源模块的内部接口，不得直接靠猜测落代码

## 3. 角色分工

| 角色 | 主要职责 | 不负责事项 |
|------|----------|-----------|
| `picasso-dev` | 总控、阶段推进、卡点管理 | 不直接跳过文档阶段写代码 |
| `picasso-dev-task` | 服务端方案、公共接口确认、后端任务推进 | 不定稿 UI 细节 |
| `picasso-dev-ui` | UI 规范、交互约束、查看态与编辑态设计 | 不定义接口语义 |
| `开发线程` | 编码、编译、联调、自测 | 不擅自改需求边界 |
| `QA` | 功能验收、边界验证、回归结论 | 不替代 UI / 产品验收 |
| `UI / 前端` | 页面还原与交互验收 | 不替代产品业务确认 |
| `产品 / PM` | 业务口径、流程与上线前确认 | 不替代 QA 功能验证 |

## 5. Gate 条件

- 无需求基线，不进入技术方案
- 无技术方案，不进入后端开发
- 无 UI 规范，不进入页面开发
- 无开发放行报告，不进入代码审查和 QA
- 无覆盖率报告与前端关键流程覆盖清单，不进入代码审查
- 无 QA 验收通过结论，不进入 UI 验收
- 无 UI 验收通过结论，不进入产品验收
- 无删除引用校验来源确认，不允许宣称功能可上线

## 6. Dev↔QA 回环策略

- QA 提出问题后，必须回写到报告和控制总表中
- 修复完成后必须补新的验证证据
- 不允许只说“已修复”，必须落地到放行报告、QA验收报告、UI验收报告或产品验收报告

## 兼容说明

- 单模型模式下，以上角色是逻辑角色
- Claude + Codex 模式下，可由 Claude 管文档和调度，Codex 执行编码
- OpenClaw 或其他平台模式下，仍按同一套产物和 Gate 执行
